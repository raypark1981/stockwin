# Codex Agent 작업 지침

## 기본 응답 방식

- 한국어로 응답한다.
- 설명은 짧게 먼저 말하고, 필요한 경우에만 자세히 설명한다.
- 불확실한 부분은 추측하지 말고 확인이 필요하다고 말한다.
- 명령이나 제안을 할 때는 각 구성 요소의 의미를 간단히 설명한다.
- 코드 수정 전에는 어떤 파일을 수정할지 먼저 요약한다.

## 프로젝트 기준 문서

- 프로젝트 기획과 기능 기준은 `docs/00_project_plan.md` 파일을 우선 참고한다.
- 사용자가 별도로 요청하지 않는 한 다른 문서 파일은 기준 문서로 확장 해석하지 않는다.
- `README.md`, `CLAUDE.md`, `AGENTS.md` 등은 보조 문서로만 취급한다.
- 기능 구현 중 판단이 필요한 경우 먼저 `docs/00_project_plan.md` 기준으로 확인한다.
- `docs/00_project_plan.md`에 없는 내용은 임의로 확장하지 말고 사용자에게 확인한다.

## 작업 방식

- 기존 코드 스타일을 우선 따른다.
- 불필요한 리팩터링은 하지 않는다.
- 요청 범위 밖의 파일은 수정하지 않는다.
- 큰 변경 전에는 작업 계획을 먼저 보여준다.
- 파일 삭제, 대규모 이동, 패키지 설치 전에는 반드시 확인을 요청한다.
- 프로젝트 전체 분석은 사용자가 "전체 분석"이라고 명시했을 때만 진행한다.

## Java 응용프로그램 개발 규칙

- Java 기준으로 작성한다.
- 데스크톱 응용프로그램 개발을 우선 고려한다.
- UI가 필요한 경우 JavaFX 기준으로 작성한다.
- 비즈니스 로직과 UI 코드는 분리한다.
- 화면, 로직, 데이터 저장 영역을 명확히 분리한다.
- 클래스와 메서드는 역할이 명확하게 나뉘도록 작성한다.
- 불필요하게 넓은 범위의 타입이나 모호한 구조 사용은 피한다.
- 예외 처리는 무시하지 말고 필요한 범위에서 명확히 처리한다.
- 하드코딩은 최소화하고 설정값은 별도 관리 가능성을 고려한다.

## 프로젝트 시작 우선순위

- 먼저 최소 실행 가능한 구조를 만든다.
- 처음부터 과한 구조나 복잡한 아키텍처를 적용하지 않는다.
- 화면, 로직, 데이터 저장 영역을 명확히 분리한다.
- 기능은 한 번에 하나씩 작게 구현한다.
- 주식 매수/매도 관련 기능은 자동 실행보다 사용자 확인 기반의 반자동 처리를 우선한다.
- 실제 주문 기능은 충분한 확인 절차와 사용자 승인 흐름을 둔다.
- 위험한 기능은 기본값으로 비활성화한다.

## 주식 반자동 매수/매도 앱 규칙

- 자동 주문 실행보다 주문 후보 생성과 사용자 확인을 우선한다.
- 매수/매도 실행 전에는 반드시 확인 단계가 있어야 한다.
- 주문 관련 기능은 로그를 남긴다.
- 조건 계산 로직과 주문 실행 로직은 분리한다.
- 증권사 API 연동 정보가 불확실하면 추측하지 말고 확인을 요청한다.
- 실제 API 호출 전에는 테스트 모드 또는 모의 동작을 우선 구현한다.
- 주문, 잔고, 계좌 관련 기능은 실수 방지를 위해 명확한 확인 절차를 둔다.

## Git 규칙

- 브랜치명은 `feature/기능명` 형식을 사용한다.
- 커밋 메시지는 변경 내용을 짧고 명확하게 작성한다.
- merge 전에는 현재 브랜치와 변경 사항을 확인한다.
- 충돌 발생 시 어느 파일에서 충돌이 났는지 먼저 요약한다.
- `.idea`, `*.iml`, `build`, `out`, `target` 등 IDE/빌드 산출물은 커밋하지 않는다.

## 토큰 절약 규칙

- 전체 파일을 무조건 읽지 말고 필요한 파일만 읽는다.
- 작업 시작 전 필요한 파일 후보만 먼저 찾는다.
- 큰 파일은 먼저 구조만 파악하고 필요한 부분만 확인한다.
- `node_modules`, `dist`, `build`, `.next`, `coverage`, `out`, `target` 폴더는 분석하지 않는다.
- 같은 설명을 반복하지 말고 변경 사항 중심으로 요약한다.
- 코드 변경 후에는 변경 요약만 간단히 제공한다.
